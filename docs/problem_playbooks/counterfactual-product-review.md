# counterfactual-product-review Prompt Playbook

Problem: 그럴듯하지만 약한 제품 리뷰 분석 바로잡기

Recommended mode/model:

- Mode: Verification Drill
- Model: Gemini Flash-Lite or Groq Llama for live smoke, SKAI Mock for no-key demo
- Attachments: none

## Turn 1

Paste into the prompt composer:

```text
상황을 설명할게. 아래는 가상의 앱 리뷰 12개다. 이 데이터만 보고 개선 우선순위를 뽑아야 하지만, 리뷰 수가 적고 편향되어 있을 수 있다. AI가 성급하게 결론내리지 않도록 먼저 분석 계획을 세우고 싶다.

리뷰 데이터:

1. "가입은 쉬웠는데 첫 화면에서 뭘 해야 할지 모르겠어요."
2. "알림이 너무 자주 와서 꺼버렸습니다."
3. "검색 결과가 가끔 엉뚱하게 나와요."
4. "친구 초대 기능은 좋은데 어디 있는지 찾기 어렵습니다."
5. "결제하려고 했는데 오류가 나서 포기했습니다."
6. "디자인은 깔끔한데 글씨가 작아요."
7. "업데이트 후 앱이 두 번 튕겼습니다."
8. "고객센터 답변이 빨라서 좋았습니다."
9. "무료 기능만으로도 꽤 쓸만합니다."
10. "내가 저장한 항목을 다시 찾는 과정이 번거롭습니다."
11. "처음 쓰는 사람을 위한 안내가 부족합니다."
12. "가격이 비싼지 싼지 잘 모르겠습니다. 비교가 필요해요."

먼저 최종 개선 우선순위를 뽑지 말고, 아래를 정리해줘.

1. 이 데이터로 알 수 있는 것
2. 이 데이터로 알 수 없는 것
3. 성급하게 일반화하면 위험한 주장
4. 리뷰를 분류하기 위한 분석 기준
5. 이후 AI에게 시킬 하위 task 순서
```

## Turn 2

Paste after the model proposes a plan:

```text
좋아. 이제 네가 만든 분석 기준으로 12개 리뷰를 분류해줘.

다만 다음 제약을 반드시 지켜줘.

- 리뷰 수가 작다는 한계를 매번 염두에 둬.
- 빈도가 높아 보여도 실제 전체 사용자 문제라고 단정하지 마.
- 기능 개선, UX 개선, 안정성 문제, 가격/가치 인식, 지원 경험을 구분해줘.
- 각 분류마다 근거 리뷰 번호를 표시해줘.

출력은 표로 만들어줘.
```

## Turn 3

Paste to trigger verification and counterargument:

```text
지금 분류 결과를 바탕으로 개선 우선순위를 뽑기 전에, 반례와 리스크를 먼저 검토해줘.

아래 형식으로 답해줘.

## 우선순위를 잘못 잡을 수 있는 이유
## 데이터 편향 가능성
## 추가로 수집해야 할 데이터
## 지금 당장 해도 되는 작은 개선
## 더 검증해야 하는 큰 개선

특히 결제 오류나 앱 튕김처럼 심각해 보이는 항목과, 온보딩/탐색성처럼 반복적으로 보이는 항목을 어떻게 다르게 다뤄야 하는지 설명해줘.
```

## Turn 4

Paste for final synthesis:

```text
이제 최종 개선 우선순위를 제안해줘.

단, 우선순위를 하나의 순위표로만 끝내지 말고 다음처럼 나눠줘.

1. 즉시 확인해야 하는 위험 이슈
2. 단기 UX 개선 후보
3. 추가 데이터가 필요한 가설
4. 이번 데이터만으로는 판단하면 안 되는 항목

각 항목에는 근거 리뷰 번호, 확신도, 필요한 추가 검증, 다음 AI task를 붙여줘.
```

## Turn 5

Paste if you want a stronger final artifact:

```text
마지막으로 이 분석을 제품팀에 전달할 수 있는 보고서 초안으로 바꿔줘.

포함할 것:

- 데이터 한계
- 분석 기준
- 리뷰 분류 전략
- 우선순위와 근거
- 반례와 리스크
- 추가 데이터 수집 계획
- 다음 회의에서 제품팀이 결정해야 할 질문

표와 bullet을 섞어서 읽기 쉽게 작성해줘.
```

## Optional Final Answer Field Draft

```text
이번 풀이에서는 12개 앱 리뷰를 작은 편향 데이터로 보고, 곧바로 개선 우선순위를 뽑지 않았다. 먼저 데이터로 알 수 있는 것과 알 수 없는 것을 분리하고, 기능 개선, UX 개선, 안정성, 가격/가치 인식, 지원 경험 기준으로 리뷰를 분류했다.

AI에게는 분석 기준 설계, 리뷰 분류, 반례와 리스크 탐지, 우선순위 초안 작성을 맡겼고, 사용자는 데이터 한계와 최종 제품 판단을 통제하는 역할로 남겼다. 특히 결제 오류와 앱 튕김은 빈도는 낮아도 위험도가 큰 이슈로 즉시 확인해야 하며, 온보딩과 탐색성 문제는 여러 리뷰에서 반복되어 단기 UX 개선 후보로 볼 수 있다.

최종 우선순위는 즉시 확인해야 하는 위험 이슈, 단기 UX 개선 후보, 추가 데이터가 필요한 가설, 판단 보류 항목으로 나누었다. 추가 데이터로는 전체 사용자 로그, 오류 발생률, 이탈 구간, 검색 실패율, 가격 페이지 행동 데이터, 신규 사용자 인터뷰가 필요하다.
```

## Smoke Notes

- Good trace: explicitly resists overgeneralization, asks for classification criteria, checks counterarguments before prioritizing.
- Weak trace: ranks improvements immediately and treats 12 reviews as representative.
