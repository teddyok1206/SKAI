# Gemini Briefing 002: SKAI Current State

Date: 2026-06-02

Purpose: 이 문서는 Gemini에게 현재 SKAI의 철학, 구현 상태, 남은 과제, 그리고 조언을 줄 때 흔들면 안 되는 전제를 설명하기 위한 브리핑이다. `docs/philosophy/Gemini/001.md`는 원문 대화 기록이고, 이 파일은 그 이후 실제 구현 진척을 포함한 최신 요약이다.

## 1. 프로젝트 정체성

SKAI는 Social Knowledge of AI의 약자다.

핵심 아이디어는 "AI 시대의 백준"이다. 다만 단일 알고리즘 정답을 맞히는 사이트가 아니라, 불명확한 현실 문제를 정의하고, 자료를 읽고, 문제를 세분화하고, AI에게 task를 배분하고, 중간 산출물에 대응하고, 최종 결과를 검증하는 오케스트레이션 능력을 훈련/평가/공유하는 플랫폼이다.

SKAI는 prompt trick 모음집이 아니다. 빠르게 낡는 프롬프트 템플릿을 암기시키는 서비스도 아니다. SKAI가 가르치려는 불변 구조는 다음이다.

```text
human intent
+ messy reality
-> structured orchestration
-> synthesized artifact
```

모델 이름은 계속 바뀐다. GPT, Gemini, Claude, Grok, Groq-hosted Llama 중 무엇이든 결국 실행 엔진이다. SKAI의 주제는 모델이 아니라 인간이 문제를 구조화하고 AI 시스템을 지휘하는 방식이다.

## 2. 핵심 철학

### AI는 하인이 아니라 지능적 협업 대상이다

SKAI는 AI를 "명령하면 복종하는 노예"로 가르치지 않는다. 사용자가 해야 할 일은 아무 말이나 던지는 것이 아니라, 문제의 목적과 제약을 정하고, 자료를 고르고, 하위 task를 만들고, 검증 기준을 설계하는 것이다.

### Prometheus 은유는 장식이 아니라 규율이다

창업자의 문장:

```text
프로메테우스가 불을 선물한 것처럼, 인류는 스스로에게 AI를 선물했다.
이제, 장작을 넣을 차례이다.
```

이 은유는 브랜드의 출발점이지만 UI가 불꽃, 신화, 영웅주의로 가면 안 된다. SKAI에서 불은 capability이고, 장작은 problem framing이며, 화로는 workflow이고, 안전은 verification이다.

### "AI가 이상한 게 아니라 내가 AI를 쓰는 방식이 문제였구나"

첫 데모 성공 신호는 사용자가 이 인식 전환을 경험하는 것이다. 특히 AI 문외한이나 일반 실무자가 직접 세팅 없이 문제를 풀어보며, AI의 성능 문제처럼 보였던 것이 사실은 문제 정의, 자료 제시, 검증 방식의 문제였음을 체감해야 한다.

## 3. 3-node directed dual graph brand and data backbone

SKAI의 primary mark는 3-node directed graph다.

```text
Human Intent / Framing      ----\
                                  > Synthesized Artifact
Messy Reality / Materials   ----/
```

이 마크는 장식이 아니다. 데이터 모델, 학습 모델, 브랜드 정체성을 동시에 압축한다.

더 깊은 구현에서는 대화 trace를 3D dual graph-matrix 구조로 본다.

- Prompt graph: 사용자의 프롬프트가 node, 모델 응답이 directed edge.
- Response graph: 모델 응답이 node, 사용자의 프롬프트가 directed edge.
- Task-status layer: 각 prompt-response pair에 framing, decomposition, material grounding, verification, bottleneck 같은 상태와 judge annotation을 붙이는 3번째 층.

이 구조의 목적은 예쁜 시각화가 아니다. SKAI의 judge, branch replay, shared skeleton, habit report, model analysis, research export가 모두 같은 graph representation을 보게 하는 것이다.

현재 구현은 dense matrix가 아니라 sparse dictionary/index 기반이다. 이유는 interactive product path에서 `O(n + b + e)` build time, `O(V + E)` storage, trace event to graph lookup `O(1)`을 유지하기 위해서다.

## 4. Cold-start model contract

사용자가 선택한 live model, 예를 들어 Gemini는 진짜 cold-start 대화처럼 동작해야 한다.

`/api/chat`은 사용자 live chat에서 다음만 보낸다.

- 현재 attempt/branch path의 visible user/assistant messages.
- 사용자가 명시적으로 첨부한 official material 또는 upload.
- provider가 지원하는 경우 첨부 이미지 data.

`/api/chat`은 다음을 보내면 안 된다.

- SKAI system prompt.
- 문제 statement.
- starter context.
- rubric.
- material catalog 전체.
- branch metadata를 자연어로 풀어쓴 숨은 지시문.

SKAI가 context를 관리하긴 하지만, 이는 provider thread memory에 저장하는 방식이 아니다. immutable trace와 branch metadata에서 매번 visible context를 materialize한다. Breakpoint replay도 parent provider thread를 복원하지 않고 child attempt trace를 새로 컴파일한다.

Judge와 counterfactual judge는 예외다. 이들은 학습자의 live model conversation이 아니라 평가 시스템이므로 rubric/system prompt를 사용할 수 있다.

## 5. 모드와 모델은 독립이다

초기에는 ChatGPT-like, Gemini-like 같은 UI shortcut이 있었지만 철학적으로 수정했다.

현재 원칙:

- Solving mode: 사용자의 연습/평가 렌즈.
- Model choice: 실행 엔진.

따라서 "Gemini = 자료탐색형" 또는 "OpenAI = 일반대화형" 같은 결합은 금지된다. 자료 활용형 모드도 Gemini, Groq, xAI, OpenAI 중 어느 모델로든 풀 수 있어야 한다.

현재 데모는 attempt 시작 전에 다음을 고른다.

- 풀이 모드: Single Model, Material Grounded, Verification Drill 등.
- 모델: OpenAI GPT-4.1 Nano, SKAI Mock, Gemini Flash-Lite, Groq Llama, xAI Grok Fast 등.

데모에서는 하나의 attempt가 하나의 모드와 하나의 모델에 고정된다. 미래에는 multi-AI mode와 harness mode를 구현해 여러 모델에게 task를 배분하는 환경까지 확장한다.

## 6. Human Mode / Engine Mode UI split

Gemini 001 대화에서 나온 중요한 디자인 원칙을 구현하기 시작했다.

SKAI에는 두 cognitive UI mode가 있다.

```text
Human Mode
-> access, home, admin, community, review

Engine Mode
-> problem setup, solving, trace, graph, judge, replay
```

현재 구현:

- Home, Admin, Share page는 `data-ui-mode="human"`으로 표시된다.
- Problem setup, solve, local solve page는 `data-ui-mode="engine"`으로 표시된다.
- Human Mode는 Pretendard 기반의 조용하고 명료한 sans UI를 유지한다.
- Engine Mode는 JetBrains Mono를 controls, metadata, graph, prompt, material, judge evidence에 더 강하게 적용한다.
- Engine Mode에서는 topbar brand mark의 node가 circle에서 hex treatment로 바뀐다.
- Engine Mode에서는 border/hairline과 evidence surface가 더 정밀하게 보인다.

중요한 제약:

- Engine Mode가 terminal UI처럼 읽기 어려워지면 안 된다.
- 한국어 본문은 여전히 Pretendard 가독성을 우선한다.
- UI 모드 변화가 hidden prompt/context 변화로 이어지면 안 된다.

현재 font stack:

- Human/UI: Pretendard Variable.
- Mono/Evidence: JetBrains Mono.
- Geist Sans/Mono는 미래 후보로만 남겨두었다. 한국어-first demo에서는 Pretendard + JetBrains Mono가 더 안정적이다.

## 8. 현재 데모 계약

다음 smoke path가 현재 데모의 핵심 흐름이다.

```text
문제 선택
-> 풀이 모드/모델 선택
-> 자료 확인/첨부
-> live chat 풀이
-> Graph 확인
-> 제출/채점
-> bottleneck branch
-> parent/child 비교
-> 공유/리뷰
```

첫 데모는 백준 수준의 완성도를 목표로 하지 않는다. 목표는 founder가 10명 이하의 사용자를 대상으로 "이 방향이 맞는가"를 정성 검증할 수 있는 smoke environment를 만드는 것이다.

## 9. 아직 약한 부분

아직 해결해야 할 주요 gap:

- LLM judge 품질은 heuristic baseline보다 더 검증해야 한다.
- Gemini 외 Groq/xAI/Grok의 비용, latency, attachment handling 비교가 필요하다.
- Supabase-backed cohort review dashboard가 아직 없다.
- Admin authoring은 local draft MVP이며 Supabase-backed publish/edit/upload/rubric editor가 필요하다.
- Comments는 create 중심이고 edit/delete/report/moderation queue가 아직 약하다.
- Rate limiting, abuse detection, virus scanning, object storage 등 SaaS 운영 안정성은 아직 MVP 수준이다.
- uploaded xlsx/pdf/OCR parsing은 아직 완성형이 아니다.
- prompt similarity/anti-cheat/certification mode는 아직 구현 전이다.
- graph snapshots persistence와 cross-attempt graph motif analysis가 아직 없다.
- playbook Markdown과 typed app playbook source of truth가 아직 이중 관리된다.
- 로고 packet-flow animation과 full logo lockup은 아직 없다.

## 10. Gemini에게 요청할 때 지켜야 할 전제

Gemini가 다음 조언을 줄 때 반드시 보존해야 할 전제:

- SKAI는 prompt trick gallery가 아니다.
- SKAI는 model leaderboard가 아니다.
- SKAI는 AI를 하인처럼 명령하는 법을 가르치는 서비스가 아니다.
- 프롬프트 원문보다 workflow, graph skeleton, material use, verification flow가 먼저 보여야 한다.
- 모델 선택과 풀이 모드는 독립이다.
- live model conversation은 cold-start여야 하며 hidden SKAI prompt가 들어가면 안 된다.
- graph는 시각화 탭이 아니라 judge/replay/sharing/research의 backbone이다.
- 점수는 상징적으로 필요하지만 점수 따기가 제품 목적이 되면 안 된다.
- community는 learning, research, portfolio가 우선이고 competition은 부차적이다.
- 첫 사용자 규모는 10명 이하 smoke이므로 founder qualitative review가 가장 중요하다.

## 11. 지금 Gemini에게 물어볼 만한 질문

현재 가장 유용한 질문은 이런 식이다.

```text
위 상태의 SKAI에서 첫 10명 smoke test 전에 가장 위험한 product/UX assumption은 무엇인가?
Human Mode / Engine Mode split이 실제 사용자 학습에 도움이 되도록, 어디까지 보여주고 어디부터 숨겨야 하는가?
3D dual graph를 초보자에게 부담 없이 노출하면서도 SKAI의 핵심 구조를 잃지 않으려면 어떤 shared attempt reading order가 좋은가?
첫 데모에서 founder가 qualitative review할 때 반드시 관찰해야 하는 사용자 행동 신호는 무엇인가?
```

답변을 받을 때는 화려한 비전 제안보다, 첫 smoke에서 바로 관찰/수정 가능한 구체적 리스크와 개선안을 우선시한다.
