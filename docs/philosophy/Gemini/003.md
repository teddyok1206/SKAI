# Gemini Briefing 003: SKAI Updated Demo State

Date: 2026-06-02

Purpose: 이 문서는 Gemini에게 SKAI의 최신 상태를 설명하기 위한 브리핑이다. 특히 `docs/philosophy/Gemini/002.md` 이후 실제로 진전된 부분을 강조한다. 이 문서를 받은 Gemini는 비전 차원의 일반 조언보다, 첫 10명 smoke test 전에 바로 점검해야 할 product/UX/evaluation 리스크를 중심으로 답해야 한다.

## 1. 한 줄 요약

SKAI는 이제 단순 아이디어나 철학 문서가 아니라, 33개 문제를 탐색하고, 문제를 선택하고, 풀이 모드와 모델을 고르고, 자료를 첨부해 cold-start live model과 대화하고, trace를 graph로 보고, judge report와 branch replay까지 수행할 수 있는 Next.js demo가 되었다.

핵심 변화:

```text
3 seed problems
-> 30 generated problems ingested
-> searchable/filterable/curated problem browser
-> graph-first sharing artifact
-> founder cohort review path
```

## 2. 프로젝트 정체성 유지

SKAI는 Social Knowledge of AI의 약자다.

SKAI는 "AI 시대의 백준"을 지향하지만, 전통적 알고리즘 정답을 맞히는 사이트가 아니다. 사용자가 불명확한 현실 문제를 정의하고, 자료를 읽고, 하위 task를 만들고, AI에게 배분하고, 중간 산출물에 대응하고, 최종 결과를 검증하는 능력을 훈련/평가/공유하는 플랫폼이다.

SKAI가 가르치려는 불변 구조:

```text
human intent
+ messy reality / materials
-> structured orchestration
-> synthesized artifact
```

절대 흔들면 안 되는 전제:

- SKAI는 prompt trick gallery가 아니다.
- SKAI는 model leaderboard가 아니다.
- SKAI는 AI에게 명령하는 법만 가르치는 서비스가 아니다.
- 모델은 실행 엔진이고, 사용자의 문제 구조화와 검증 흐름이 평가 대상이다.
- 점수는 필요하지만 score grinder가 되면 안 된다.

## 3. 002 이후 가장 큰 진전

### 3.1 문제 수가 3개에서 33개로 늘었다

기존 seed problem 3개에 더해, ChatGPT Pro로 생성한 30문제 batch 001을 검토하고 앱에 반영했다.

원본 batch:

- `docs/problem_generation/batches/001_30problems.json`

앱용 정규화 모듈:

- `data/generated-problem-batch-001.ts`

반영 방식:

- 생성 원본은 archive로 보존했다.
- 앱에서는 typed normalization layer를 통해 `Problem`과 `ProblemPlaybook`으로 변환한다.
- `rubricKey: "defaultRubric"`은 실제 `defaultRubric`으로 변환한다.
- 생성된 `recommendedMode`는 현재 SKAI 모드 label로 정규화했다.
- `/synthetic/skai/...` 형태의 가짜 material href는 앱에서 제거했다. 실제 파일이 없는데 다운로드/이미지 링크가 생기는 운영 불안정성을 피하기 위해서다.
- material의 `extractedText`는 유지해서 AI context와 material viewer에서 사용할 수 있게 했다.

Batch 001 coverage:

- category: workplace 5, research 5, creative 5, data_analysis 5, coding 5, strategy 5
- difficulty: intro 8, standard 14, advanced 8
- goalProfile: accuracy_first 5, speed_first 5, cost_constrained 5, workflow_adoption 5, learning_oriented 5, exploration 5
- material kinds: text 52, spreadsheet 9, image 1, pdf 4, csv 12

자동 검토 결과:

- duplicate id 없음.
- app enum mismatch 없음.
- material extractedText 누락 없음.
- playbook attachment id mismatch 없음.
- 명백한 phone/API-key-like sensitive string 없음.

중요한 의미:

SKAI는 이제 "문제풀이 플랫폼이 될 수 있다"는 말만 하는 상태가 아니라, 다양한 domain과 난이도의 문제를 실제로 보여주고 고르게 할 수 있다.

### 3.2 문제 탐색 UI가 생겼다

33개 문제가 되면서 flat grid는 더 이상 충분하지 않았다. 그래서 홈 문제 목록을 `ProblemBrowser`로 교체했다.

구현:

- `components/problem-browser.tsx`

기능:

- keyword search
- category filter
- difficulty filter
- goal profile filter
- material kind filter
- curation lane
- result count
- top domain summary
- reset action
- empty state

검색 대상:

- problem id
- title/subtitle
- statement
- userGoal
- constraints
- starterContext
- deliverables
- material title/description/kind/fileName/extractedText
- generated classification metadata
- primaryFailureMode
- idealBottleneck
- graphPattern

현재 curation lanes:

- 전체
- 빠른 시작
- 자료 중심
- 검증 연습
- 심화 replay
- 문제정의

중요한 설계 결정:

문제 탐색 축은 model/provider가 아니다. 분야, 난이도, 목표, 자료, 검증/문제정의/자료활용 같은 orchestration practice axis를 중심으로 한다. 모델 선택은 문제 진입 후 attempt 시작 전 단계에서만 이뤄진다.

### 3.3 공유 화면이 점수표보다 SKAI Artifact를 먼저 보여준다

`002.md` 시점에는 graph skeleton과 sharing direction은 있었지만, 이후 더 명확한 artifact surface가 들어갔다.

구현된 개념:

- SKAI Artifact card
- SVG export
- Universal Reading Layer
- Graph Transition Log
- Intelligence Mirror

의미:

공유 화면은 "몇 점 받았는가"보다 "이 사용자가 어떤 directed orchestration timeline을 만들었는가"를 먼저 보여준다. 초보자는 graph 용어를 몰라도 problem/material/process/checking layer를 통해 읽을 수 있고, 심화 사용자는 graph skeleton과 raw transcript를 열어볼 수 있다.

이 방향은 SKAI가 prompt 원문 공유 사이트가 아니라, 문제 접근 구조를 공유하는 커뮤니티로 가기 위해 중요하다.

### 3.4 Intelligence Mirror가 score report 앞단에 들어갔다

Score report는 이제 단순 채점표가 아니라 사용자의 문제 구조화 능력을 비추는 mirror로 시작한다.

Mirror가 보는 것:

- intent
- control
- verification
- artifact formation

의미:

SKAI가 평가하는 것은 "AI가 낸 답"만이 아니다. 사용자가 어떻게 문제의 목적을 붙잡았고, 자료와 제약을 통제했으며, 검증했고, 최종 artifact를 만들었는지를 먼저 보여준다.

### 3.5 댓글 lifecycle이 보강됐다

공유 풀이의 prompt skeleton 카드에는 trace event별 댓글과 답글이 있다. 이후 여기에 다음 lifecycle이 추가됐다.

- edit
- soft-delete
- report
- Supabase owner-only update policy
- report table and count trigger

의미:

prompt-level discussion은 SKAI 커뮤니티의 핵심이 될 수 있다. 하지만 학습 맥락을 보존해야 하므로 hard delete 중심이 아니라 thread structure를 유지하는 soft-delete/report 구조가 더 적합하다.

### 3.6 Founder cohort review가 Supabase까지 확장됐다

Admin founder dashboard는 localStorage smoke attempts뿐 아니라 Supabase remote cohort snapshot도 볼 수 있다.

중요한 안전장치:

- service role 전체 cohort 조회는 `SKAI_FOUNDER_EMAILS` allowlist가 있을 때만 허용한다.
- 그 외에는 현재 사용자 RLS scope로 제한된다.

의미:

첫 10명 smoke test에서 founder가 DB를 직접 열지 않고도 사용자 풀이의 문제/모델/모드/점수/비용/branch 상태를 정성 검토할 수 있다.

### 3.7 Brand/UI 정체성이 더 명확해졌다

Gemini 001에서 논의한 brand direction을 구현으로 더 흡수했다.

구현:

- primary topbar mark는 flame이 아니라 3-node directed graph.
- Human Mode / Engine Mode split.
- Engine Mode에서 graph node가 hex treatment로 변환.
- sparse packet-flow animation.
- reusable SKAI logo lockup.

Logo lockup:

- 3-node mark
- SKAI wordmark
- Social Knowledge of AI full name

중요한 방향:

Prometheus/fire는 origin metaphor이지만 primary icon system이 아니다. SKAI의 primary mark는 intent/material -> artifact를 나타내는 directed graph다.

## 5. 현재 smoke contract

현재 첫 데모가 증명해야 하는 flow:

```text
문제 탐색
-> 문제 선택
-> 풀이 모드/모델 선택
-> 자료 확인/첨부
-> live chat 풀이
-> Graph 확인
-> 제출/채점
-> bottleneck branch
-> parent/child 비교
-> 공유/리뷰
-> founder qualitative review
```

이번 업데이트 이후 smoke에서 특히 관찰해야 하는 새 행동:

- 사용자가 33개 문제 중 어떤 기준으로 문제를 고르는가?
- 검색/필터/curation lane이 실제로 도움 되는가, 아니면 선택 피로를 만드는가?
- "자료 중심", "검증 연습", "문제정의" lane이 사용자의 학습 목표와 맞는가?
- generated problem들이 실제 풀이 가능한가, 아니면 겉보기만 그럴듯한가?
- playbook이 cold-start live model에게 충분한 context를 visible prompt로 전달하는가?
- material extractedText만 있는 synthetic 자료가 학습 경험을 방해하지 않는가?

## 6. 아직 약한 부분

현재 주요 gap:

- LLM judge 품질은 heuristic baseline보다 더 검증해야 한다.
- `SKAI_JUDGE_MODE=llm` 또는 `ensemble`로 실제 judge provider 비교가 필요하다.
- Gemini 외 Groq/xAI/OpenAI cheap baseline의 latency, cost, attachment handling 비교가 필요하다.
- OpenAI cheap baseline으로는 `gpt-5-nano`를 붙이는 것이 후보지만 아직 코드 반영 전이다.
- Problem browser는 local static data 기준이다. Supabase-backed problem discovery, saved filters, personalized recommendation은 없다.
- Generated problem 30개는 자동 검토를 통과했지만 실제 사람이 풀어본 품질 검증은 아직 없다.
- Generated problem에는 synthetic href가 제거되어 extractedText 중심으로만 자료를 본다. 실제 binary asset generation/import는 아직 없다.
- Admin authoring은 local draft MVP다. Supabase-backed authoring, multi-material upload, rubric editor, playbook editor가 필요하다.
- Founder remote notes/export/advanced filtering은 아직 부족하다.
- Rate limiting, abuse detection, virus scanning, object storage는 아직 production 수준이 아니다.
- uploaded xlsx/pdf/OCR parsing은 MVP 밖이다.
- certification/anti-cheat/prompt similarity는 아직 구현 전이다.
- graph snapshot persistence, cross-attempt motif analysis, habit report는 아직 후속 과제다.
- playbook Markdown과 typed app playbook은 여전히 source of truth가 이중화되어 있다.

## 7. Gemini가 조언할 때 지켜야 할 기준

Gemini는 다음 방향을 흔들면 안 된다.

- 모델 비교보다 human orchestration assessment가 중심이다.
- 문제 탐색도 provider/model이 아니라 practice situation을 기준으로 해야 한다.
- live model은 hidden SKAI context 없이 cold-start여야 한다.
- Graph는 시각화 장식이 아니라 judge/replay/sharing/research backbone이다.
- 공유 화면은 score/transcript보다 artifact/workflow를 먼저 보여줘야 한다.
- 초보자에게 graph terminology를 강요하지 않되, 구조적 사고를 숨기면 안 된다.
- Generated problem을 많이 만드는 것은 중요하지만, 문제 품질과 real-world ambiguity가 더 중요하다.
- 첫 smoke는 founder qualitative review가 핵심이다.

## 8. 지금 Gemini에게 묻고 싶은 질문

아래 질문들에 답해달라. 가능하면 추상적 비전보다, 첫 10명 smoke 전에 바로 반영하거나 관찰할 수 있는 구체적 항목으로 답해달라.

### Question 1: 33문제 problem browser의 가장 큰 UX 리스크

현재 문제 탐색은 검색, category/difficulty/goal/material filter, curation lane으로 구성되어 있다. 첫 사용자에게 이 구조가 선택을 도울까, 아니면 부담을 줄까? 어떤 정보를 줄이고 어떤 정보를 남겨야 하는가?

### Question 2: generated problem quality review 기준

ChatGPT Pro가 만든 30개 문제는 schema 검토를 통과했다. 하지만 실제 SKAI 문제로 좋은지 판단하려면 어떤 editorial checklist가 필요한가? 특히 "겉보기만 현실적이고 실제로는 깊은 orchestration을 요구하지 않는 문제"를 어떻게 걸러야 하는가?

### Question 3: curation lane 재설계

현재 lane은 전체, 빠른 시작, 자료 중심, 검증 연습, 심화 replay, 문제정의다. SKAI의 철학상 더 나은 lane 이름이나 구조가 있는가? 이 lane들이 사용자의 학습 목표와 문제의 goalProfile/primarySkill을 잘 연결하는가?

### Question 4: 첫 smoke에서 founder가 봐야 할 행동 신호

사용자가 문제를 고르는 순간부터 공유/리뷰까지, founder가 반드시 기록해야 하는 qualitative signal은 무엇인가? 특히 "AI가 이상한 게 아니라 내가 쓰는 방식이 문제였구나"라는 인식 전환을 어떻게 관찰할 수 있는가?

### Question 5: 다음 slice 우선순위

현재 다음 후보는 다음이다.

- OpenAI cheap baseline `gpt-5-nano` 추가.
- LLM judge calibration.
- generated problem editorial dashboard.
- problem browser saved curation / better filters.
- Supabase-backed problem authoring.
- graph snapshot persistence.
- founder review export.

첫 10명 smoke 전 무엇을 먼저 해야 하는가? 이유와 최소 구현 범위를 제안해달라.

## 9. 답변 형식 요청

아래 형식으로 답해달라.

```text
1. Highest-risk assumption
2. Problem browser critique
3. Generated problem editorial checklist
4. Smoke test observation protocol
5. Recommended next 3 implementation slices
6. Things SKAI should explicitly avoid
```

각 항목은 가능한 한 구체적으로 작성해달라. "비전을 확장하라"보다 "첫 smoke에서 무엇을 바꿔보고 무엇을 관찰할지"를 우선하라.
