# Gemini Response 004: Smoke Test Risk And Editorial Gate

Date: 2026-06-02

Source: 사용자가 Gemini에게 `Gemini Briefing 003`을 보내고 받은 답변을 SKAI 실행 문서로 정리한 것이다. 원문의 수사는 줄이고, 제품/기술 결정에 직접 반영할 항목만 보존한다.

## 1. Highest-Risk Assumption

가장 위험한 가정은 생성형 AI가 만든 30개 synthetic problem이 구조적으로 완벽할 것이라는 믿음이다.

핵심 리스크:

- 문제의 자료, 제약, 산출물 사이에 논리적 구멍이 있으면 사용자는 자신의 AI orchestration 방식을 의심하지 않는다.
- 대신 "SKAI 문제가 대충 만들어졌네"라고 느끼고 시스템 권위를 불신한다.
- 첫 smoke test에서 30개 generated problem을 모두 노출하면 선택 피로와 품질 리스크가 동시에 커진다.

반영 방향:

- 첫 10명 smoke에서는 seed problem 3개를 기본 노출한다.
- generated problem은 founder/editorial checklist를 통과한 것만 홈 ProblemBrowser에 노출한다.
- generated batch는 보존하되 publish gate를 둔다.

## 2. Problem Browser Critique

현재 33개 문제를 category, difficulty, goalProfile, materialKind 등으로 모두 펼쳐둔 browser는 첫 사용자에게 선택 마찰을 줄 수 있다.

반영 방향:

- 문제 탐색의 기본 표면은 3개 수준의 scenario lane 중심으로 줄인다.
- `goalProfile`, `primarySkill`, `graphPattern`, failure mode 같은 내부 메타데이터는 시작 전 카드에서 숨긴다.
- advanced filter는 필요하면 접어서 제공한다.
- 카드에는 제목, 핵심 제약 한 줄, 예상 시간 같은 즉시 판단 가능한 정보만 우선 노출한다.

## 3. Generated Problem Editorial Checklist

Generated problem은 아래 4단계 kill switch를 통과해야 smoke에 노출한다.

1. Anti-One-Shot Call
   - 한 줄로 "자료 다 읽고 최종 결과 만들어줘"라고 해도 그럴듯한 답이 나오면 탈락이다.
   - 반드시 decomposition, intermediate verification, or bottleneck이 있어야 한다.

2. Material Cross-Reference
   - 자료 하나만 읽어도 풀리면 탈락이다.
   - Material A의 기준으로 Material B의 오류나 누락을 검토해야 한다.

3. Synthetic Href Missing Check
   - 이미지/엑셀/PDF가 extracted text만으로도 불쾌하지 않게 읽혀야 한다.
   - 표 구조나 공간 맥락 유실로 풀이 자체가 어려우면 탈락이다.

4. Domain Noise Score
   - 약대생/일반 지인이 풀기 전에 과도한 전문지식 장벽에 막히면 탈락이다.
   - 오케스트레이션보다 도메인 지식이 먼저 시험되면 안 된다.

## 4. Smoke Test Observation Protocol

Founder가 기록해야 할 qualitative signal:

- Discovery:
  - 사용자가 난이도, 분야, scenario lane 중 무엇을 먼저 보는가?
  - lane 이름을 이해하지 못하면 UX vocabulary mismatch다.

- Solving and Control:
  - 첫 prompt 입력 전에 material을 읽는 시간이 있는가?
  - 자료를 보지 않고 바로 채팅창에 쓰면 AI를 여전히 answer vending machine으로 대하는 신호다.

- Aha Moment:
  - Bad signal: "AI가 별로네."
  - Good signal: "내가 조건을 빠뜨렸네.", "자료를 먼저 첨부했어야 했네."

- Mirroring:
  - Intelligence Mirror 상단을 읽는가?
  - 점수 숫자로 바로 뛰는가?

## 5. Recommended Next Slices

1. Generated Problem Editorial Dashboard
   - Admin에서 generated problem별 checklist를 보고 publish/hide를 토글한다.
   - 홈 ProblemBrowser는 seed + published generated problem만 보여준다.

2. LLM Judge Calibration and Human Score Override
   - Founder가 judge 오심을 수동 override할 수 있어야 한다.
   - 첫 smoke에서는 judge 신뢰도가 철학 신뢰도와 직결된다.

3. Step-by-Step Universal Reading Layer Animation
   - graph가 처음 보는 사람에게 이진 코드처럼 보이지 않게 prompt/response flow를 순차적으로 보여준다.
   - 단, 장식이 아니라 대화 흐름이 구조물로 쌓인다는 직관을 주는 수준이어야 한다.

## 6. Explicit Avoidance

SKAI가 피해야 할 것:

- "가장 인기 있는 모델로 풀기", "Gemini 전용 문제" 같은 provider/model 중심 탐색.
- 대화창 내부의 prompt cheat-key/template popup.
- global leaderboard를 공유 화면 상단에 두는 score grinder 구조.
- 문제 품질 검증 없이 generated problem을 대량 공개하는 것.

## Accepted Immediate Action

이번 답변을 받아들여 다음을 바로 반영한다.

- Generated problem은 기본 숨김 처리한다.
- Admin에 editorial dashboard를 추가한다.
- 4-step checklist를 통과한 generated problem만 smoke browser에 노출한다.
- ProblemBrowser 기본 UI는 scenario lane 중심으로 줄이고, 내부 메타데이터 노출을 줄인다.

## Philosophy Check

이 답변은 SKAI의 철학과 잘 맞는다. 생성 문제를 무작정 늘리는 것보다 "현실 문제의 논리적 견고함"과 "사용자의 orchestration 책임"을 지키는 쪽을 우선한다. 또한 문제 탐색이 model/provider 중심으로 흐르지 않도록 막는다.
